## Project Overview

A read-only resource scheduling calendar for convention/expo events. Displays events in two grid views (panels-as-columns, rooms-as-columns) with day/district filtering, category-based color coding, and print output. Data syncs from Airtable to PostgreSQL.

Data flow: `Airtable â†’ POST /api/sync/airtable â†’ PostgreSQL â†’ GET /api/events â†’ Frontend`

## Commands

- **Dev server:** `npm run dev` (starts Express + Vite on port 3000)
- **Type check:** `npm run check` (runs `tsc`)
- **Build:** `npm run build` (Vite frontend build + esbuild server bundle)
- **Build client only:** `npm run build:client` (Vite frontend, used by Vercel)
- **Build API only:** `npm run build:api` (esbuild bundles `server/api-entry.ts` â†’ `api/index.js`)
- **Start production:** `npm run start`

No test framework is configured.

## Architecture

### Monorepo Structure (3 layers)

- **`shared/`** â€” Drizzle schema definitions (`schema.ts`) and Zod validation schemas. Shared types (`Panel`, `Room`, `Event`) used by both client and server. Aliased as `@shared`.
- **`server/`** â€” Express.js backend. `routes.ts` defines all REST endpoints under `/api`. `storage.ts` implements `IStorage` interface with `DbStorage` class (Drizzle + Neon HTTP driver). `db.ts` configures the database connection.
- **`client/`** â€” React 18 SPA. Vite root is `client/`. Aliased as `@` pointing to `client/src/`. Uses `@assets` for `attached_assets/`.

### Key Patterns

- **Read-only app:** No CRUD UI. All data comes from Airtable sync.
- **Storage abstraction:** All DB access goes through the `IStorage` interface in `server/storage.ts`. Routes call `storage.*` methods.
- **Query keys as paths:** TanStack Query uses API paths as query keys (e.g., `queryKey: ["/api/panels"]`). The query client is configured with a default fetch function that uses these keys as URLs.
- **Schema-driven validation:** Zod schemas are auto-generated from Drizzle table definitions via `drizzle-zod`. Insert schemas omit `id` (UUID auto-generated by Postgres).
- **Two grid modes:** "Panels" shows panels as columns with events positioned by time. "Rooms" shows rooms as columns with events showing their panel name.

### Database

- PostgreSQL via Neon serverless (HTTP mode)
- Three tables: `panels`, `rooms`, `events`
  - `panels`: id (uuid PK), panel_name (text)
  - `rooms`: id (uuid PK), room_name (text), district (text, nullable)
  - `events`: id (uuid PK), title (text), event_type (text), date (text, YYYY-MM-DD), start_time (text, HH:MM), end_time (text, HH:MM), panel_id (FK â†’ panels, cascade, nullable), room_id (FK â†’ rooms, cascade, nullable)
- Requires `DATABASE_URL` env var

### API Endpoints

- `GET /api/panels` â€” all panels
- `GET /api/rooms` â€” all rooms
- `GET /api/events` â€” all events, optional `?date=YYYY-MM-DD` and `?roomId=UUID` query params
- `POST /api/sync/airtable` â€” destructive sync (clears all data, reloads from Airtable)

### Airtable Sync

- Endpoint: `POST /api/sync/airtable` (destructive â€” clears all data then reloads)
- Requires env vars: `AIRTABLE_PERSONAL_ACCESS_TOKEN`, `AIRTABLE_BASE_ID`
- Airtable tables: "Panels", "Rooms", "Events" with "Grid view"
- Sync order: clearAll â†’ Panels â†’ Rooms â†’ Events
- Builds ID maps (Airtable record ID â†’ Postgres UUID) for foreign key resolution
- Linked record fields (Panel, Room) return arrays â€” first element is extracted
- Times stored as plain strings (HH:MM format), dates as YYYY-MM-DD

### Frontend

- Single page app with one route (`/`). Wouter handles routing.
- UI components from shadcn/ui (New York style) built on Radix UI primitives, in `client/src/components/ui/`.
- `client/src/components/examples/` contains reference implementations of the old components.
- Two grid views: CalendarGrid (panels as columns) and RoomsGrid (rooms as columns)
- Calendar grid displays 8 AMâ€“10 PM time slots
- Filtering: date picker, grid mode toggle (Panels/Rooms), district filter (rooms mode only)
- Filters persisted in localStorage and URL query params (`?mode=panels&date=2025-07-24`)
- Print button triggers `window.print()` with clean print CSS
- Styling: Tailwind CSS with CSS variable theming for light/dark modes.

### Vercel Deployment

- **Hosted at:** https://resource-scheduling-calendar.vercel.app
- **Static frontend:** Vite builds to `dist/public`, served by Vercel CDN
- **Serverless API:** Single catch-all Express function at `api/index.js`
- **Build:** `npm run build:client` (Vite) + `npm run build:api` (esbuild)

**Key constraint:** Vercel auto-detects `.ts` files in `api/` and compiles them itself, but its built-in compilation cannot resolve cross-directory imports or TypeScript path aliases. The solution is to keep the source at `server/api-entry.ts`, pre-bundle it with esbuild into `api/index.js` (which inlines all `server/` and `shared/` code), and commit the built file. Vercel then serves the self-contained JS bundle directly.

```
server/api-entry.ts  â†’  esbuild  â†’  api/index.js  â†’  Vercel serves
```

After changing server code, rebuild with `npm run build:api` and commit `api/index.js`.
