## Project Overview

A read-only resource scheduling calendar for convention/expo events. Displays events in two grid views (panels-as-columns, rooms-as-columns) with day/district filtering, category-based color coding, and print output. Data syncs from Airtable to PostgreSQL.

Data flow: `Airtable → POST /api/sync/airtable → PostgreSQL → GET /api/events → Frontend`

## Commands

- **Dev server:** `npm run dev` (starts Express + Vite on port 5000)
- **Type check:** `npm run check` (runs `tsc`)
- **Build:** `npm run build` (Vite frontend build + esbuild server bundle)
- **Start production:** `npm run start`
- **Push DB schema:** `npm run db:push` (drizzle-kit push)

No test framework is configured.

## Architecture

### Monorepo Structure (3 layers)

- **`shared/`** — Drizzle schema definitions (`schema.ts`) and Zod validation schemas. Shared types (`Panel`, `Room`, `Event`) used by both client and server. Aliased as `@shared`.
- **`server/`** — Express.js backend. `routes.ts` defines all REST endpoints under `/api`. `storage.ts` implements `IStorage` interface with `DbStorage` class (Drizzle + Neon HTTP driver). `db.ts` configures the database connection.
- **`client/`** — React 18 SPA. Vite root is `client/`. Aliased as `@` pointing to `client/src/`. Uses `@assets` for `attached_assets/`.

### Key Patterns

- **Read-only app:** No CRUD UI. All data comes from Airtable sync.
- **Storage abstraction:** All DB access goes through the `IStorage` interface in `server/storage.ts`. Routes call `storage.*` methods.
- **Query keys as paths:** TanStack Query uses API paths as query keys (e.g., `queryKey: ["/api/panels"]`). The query client is configured with a default fetch function that uses these keys as URLs.
- **Schema-driven validation:** Zod schemas are auto-generated from Drizzle table definitions via `drizzle-zod`. Insert schemas omit `id` (UUID auto-generated by Postgres).
- **Two grid modes:** "Panels" shows panels as columns with events positioned by time. "Rooms" shows rooms as columns with events showing their panel name.

### Database

- PostgreSQL via Neon serverless (HTTP mode)
- Three tables: `panels`, `rooms`, `events`
  - `panels`: id (uuid PK), panel_name (text)
  - `rooms`: id (uuid PK), room_name (text), district (text, nullable)
  - `events`: id (uuid PK), title (text), event_type (text), date (text, YYYY-MM-DD), start_time (text, HH:MM), end_time (text, HH:MM), panel_id (FK → panels, cascade, nullable), room_id (FK → rooms, cascade, nullable)
- Requires `DATABASE_URL` env var

### API Endpoints

- `GET /api/panels` — all panels
- `GET /api/rooms` — all rooms
- `GET /api/events` — all events, optional `?date=YYYY-MM-DD` and `?roomId=UUID` query params
- `POST /api/sync/airtable` — destructive sync (clears all data, reloads from Airtable)

### Airtable Sync

- Endpoint: `POST /api/sync/airtable` (destructive — clears all data then reloads)
- Requires env vars: `AIRTABLE_PERSONAL_ACCESS_TOKEN`, `AIRTABLE_BASE_ID`
- Airtable tables: "Panels", "Rooms", "Events" with "Grid view"
- Sync order: clearAll → Panels → Rooms → Events
- Builds ID maps (Airtable record ID → Postgres UUID) for foreign key resolution
- Linked record fields (Panel, Room) return arrays — first element is extracted
- Times stored as plain strings (HH:MM format), dates as YYYY-MM-DD

### Frontend

- Single page app with one route (`/`). Wouter handles routing.
- UI components from shadcn/ui (New York style) built on Radix UI primitives, in `client/src/components/ui/`.
- `client/src/components/examples/` contains reference implementations of the old components.
- Two grid views: CalendarGrid (panels as columns) and RoomsGrid (rooms as columns)
- Calendar grid displays 8 AM–10 PM time slots
- Filtering: date picker, grid mode toggle (Panels/Rooms), district filter (rooms mode only)
- Filters persisted in localStorage and URL query params (`?mode=panels&date=2025-07-24`)
- Print button triggers `window.print()` with clean print CSS
- Styling: Tailwind CSS with CSS variable theming for light/dark modes.

## Environment Variables

- `DATABASE_URL` — PostgreSQL connection string (required)
- `AIRTABLE_PERSONAL_ACCESS_TOKEN` — Airtable personal access token (for sync)
- `AIRTABLE_BASE_ID` — Airtable base identifier (for sync)
- `PORT` — Server port (default: 5000)
