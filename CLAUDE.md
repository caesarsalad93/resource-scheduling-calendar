## Project Overview

A read-only resource scheduling calendar for convention/expo events. Displays events by resource (called "Panels") in a day-view grid with category-based color coding and saved view presets. Data syncs from Airtable to PostgreSQL to avoid API rate limits.

Data flow: `Airtable → POST /api/sync/airtable → PostgreSQL → GET /api/events → Frontend`

## Commands

- **Dev server:** `npm run dev` (starts Express + Vite on port 5000)
- **Type check:** `npm run check` (runs `tsc`)
- **Build:** `npm run build` (Vite frontend build + esbuild server bundle)
- **Start production:** `npm run start`
- **Push DB schema:** `npm run db:push` (drizzle-kit push)

No test framework is configured.

## Architecture

### Monorepo Structure (3 layers)

- **`shared/`** — Drizzle schema definitions (`schema.ts`) and Zod validation schemas. Shared types (`Panel`, `Event`) used by both client and server. Aliased as `@shared`.
- **`server/`** — Express.js backend. `routes.ts` defines all REST endpoints under `/api`. `storage.ts` implements `IStorage` interface with `DbStorage` class (Drizzle + Neon HTTP driver). `db.ts` configures the database connection.
- **`client/`** — React 18 SPA. Vite root is `client/`. Aliased as `@` pointing to `client/src/`. Uses `@assets` for `attached_assets/`.

### Key Patterns

- **Storage abstraction:** All DB access goes through the `IStorage` interface in `server/storage.ts`. Routes call `storage.*` methods, not `db` directly (except the sync/seed endpoints).
- **Query keys as paths:** TanStack Query uses API paths as query keys (e.g., `queryKey: ["/api/panels"]`). The query client is configured with a default fetch function that uses these keys as URLs.
- **Schema-driven validation:** Zod schemas are auto-generated from Drizzle table definitions via `drizzle-zod`. Insert schemas omit `id` (UUID auto-generated by Postgres).
- **Panel = Resource:** The domain model was renamed from "Resource" to "Panel." Legacy `Resource`/`InsertResource` types exist as aliases. Frontend still uses `resourceId` in some places, mapped to `panelId` before sending to backend.

### Database

- PostgreSQL via Neon serverless (HTTP mode, not WebSocket — more reliable)
- Two tables: `panels` and `events` (FK: `events.panel_id → panels.id` with cascade delete)
- Panel types: Autograph, Exclusive, Panel, Cart, Media
- Event categories: autograph, exclusive, panel, cart, media, miscellaneous
- Requires `DATABASE_URL` env var

### Airtable Sync

- Endpoint: `POST /api/sync/airtable` (destructive — clears all data then reloads)
- Requires env vars: `AIRTABLE_API_TOKEN`, `AIRTABLE_BASE_ID`
- Airtable tables must be named "Panels" and "Events" with "Grid view"
- Linked record fields return arrays — first element is extracted
- **Timezone quirk:** Airtable returns UTC (`Z` suffix) but times represent wall-clock times. The `Z` is stripped during sync and again on frontend display to prevent unwanted UTC conversion. See `DEBUGGING_NOTES.md` for details.

### Frontend

- Single page app with one route (`/`). Wouter handles routing.
- UI components from shadcn/ui (New York style) built on Radix UI primitives, in `client/src/components/ui/`.
- `client/src/components/examples/` contains reference implementations of the main components.
- Calendar grid displays 8 AM–7 PM time slots with panels as columns.
- Saved views stored in localStorage.
- Styling: Tailwind CSS with CSS variable theming for light/dark modes.

## Environment Variables

- `DATABASE_URL` — PostgreSQL connection string (required)
- `AIRTABLE_API_TOKEN` — Airtable personal access token (for sync)
- `AIRTABLE_BASE_ID` — Airtable base identifier (for sync)
- `PORT` — Server port (default: 5000)
